# Claude Proxy 公众号教程版设计

## 目标

把一份偏技术执行向的 `AWS + Xray + mihomo + 住宅代理` 手写教程，整理成一篇适合公众号发布的中文教程稿。

这篇文章不再走“现象拆解型”观点稿，而改成 `方法复盘型` 教程稿：按普通用户第一次搭建 Claude 通道的真实路径，把准备、搭建、连接、验证、排障这 5 段走顺，并尽量降低术语门槛。

## 目标读者

- 第一次搭这类 Claude 通道的普通用户
- 想稳定使用 Claude / Claude Code，但并不熟悉 VPS、Xray、mihomo 的读者
- 希望有一篇“能跟着做、知道哪里会卡、知道怎么判断成功”的教程，而不是纯原理说明的人

## 成功标准

这篇文章写完后，目标读者应该能得到 4 件东西：

1. 知道这套方案具体在解决什么问题，不把它误解成“万能代理教程”。
2. 知道动手前要提前准备什么，不会做到一半才发现少了关键信息。
3. 知道正确的执行顺序：服务端搭建 -> 客户端连接 -> 验证结果 -> 常见排障。
4. 知道每个关键步骤“看到什么算成功”，减少“好像装完了但不确定”的焦虑。

## 核心判断

这篇教程最重要的一句判断不是“这套架构多高级”，而是：

`第一次搭 Claude 通道，真正劝退人的通常不是命令本身，而是不知道该先准备什么、做到哪一步算成功、卡住后先查哪里。`

## 文章定位

- 主类型：`方法复盘型`
- 次定位：`新手教程 + 避坑整合`
- 主价值：让普通用户可以沿一条主线走通，而不是在配置片段里来回跳

## 写法原则

1. 默认读者不懂术语。
   第一次出现 `VLESS`、`Reality`、`Xray`、`mihomo`、`住宅代理` 时，只用一句人话解释它在整条链路里的职责，不展开底层原理。

2. 默认读者怕“不确定”。
   每个关键步骤都要写出“成功信号”，例如监听到哪个端口、返回什么结果、日志里看到什么。

3. 默认读者不需要完整网络课。
   原理说明只保留够用版本，重点放在执行顺序、关键配置、验证方法和排障入口。

4. 默认读者更需要顺序感，而不是知识全景图。
   文章必须沿单一主线推进，不写成百科式材料堆叠。

5. 默认读者会复制命令。
   代码块要尽量按动作分组，不堆过长的连续命令；每组命令前先说“这一步在做什么”，命令后说“执行完看哪里”。

## 标题方向

文章标题要同时满足两个目标：

- 让普通用户知道“这是一篇能照着做的教程”
- 让读者意识到“真正难点不在命令，而在整条路径没人讲清”

推荐标题方向：

1. `想稳定用 Claude，这条 AWS + Xray + mihomo 通道怎么搭？一篇带你从头走通`
2. `Claude 通道怎么搭？这篇把 AWS、Xray、mihomo 和住宅代理一次讲清`
3. `第一次搭 Claude 通道，最容易卡住的不是命令，而是这几步没人讲透`
4. `从 AWS 到手机连接，Claude 通道这篇按普通用户视角带你走一遍`
5. `Claude / Claude Code 稳定使用怎么做？这套通道搭建流程适合第一次动手的人`

推荐主标题：

`第一次搭 Claude 通道，最容易卡住的不是命令，而是这几步没人讲透`

## 开头设计

开头前 3 段分别承担：

1. 场景：
   先点出很多人现在不是“想偶尔试试 Claude”，而是“想稳定把 Claude 接进自己的日常工作”。

2. 判断：
   直接下判断：第一次搭这类通道，最容易卡住的不是安装命令，而是不知道整条链路该怎么顺着走。

3. 承诺：
   明确告诉读者这篇会给什么：
   - 需要提前准备的东西
   - 服务端怎么搭
   - 客户端怎么连
   - 怎么验证 Anthropic 流量真的走对了
   - 卡住先查哪里

## 正文章节骨架

### 第一部分：先说清这套方案到底在做什么

职责：
- 用一张简化链路图说明整体结构
- 让读者知道这不是“全局代理”，而是“客户端连到 VPS，VPS 再把 Claude 相关流量单独分流给住宅代理”

输出重点：
- 一句人话版解释整条链路
- 一张结构图
- 一句风险提示：这套方案适合愿意自己维护的人

### 第二部分：动手前先准备这 4 样东西

职责：
- 先把准备工作收口，降低新手中途掉链子的概率

建议写成 4 项清单：
- 一台 AWS EC2（Ubuntu）
- 一个住宅代理账号
- 一台电脑和至少一台移动设备
- 基本 SSH 登录能力

每项都要写：
- 它是做什么的
- 没有它会卡在哪
- 读者要提前记下哪些信息

### 第三部分：服务端怎么搭

职责：
- 按顺序写服务端搭建，不把 Xray 和 mihomo 混在一起

建议拆成 4 段：
- 先创建 AWS EC2 与安全组
- 安装并配置 Xray
- 安装并配置 mihomo
- 启动服务并确认端口监听正常

每段都要包含：
- 这一步在做什么
- 复制哪段命令
- 哪些字段需要替换
- 看到什么算成功

### 第四部分：客户端怎么连

职责：
- 让普通用户知道最终怎么真正用起来

建议按设备分小节：
- Mac：Clash Verge Rev
- iPhone：Hiddify
- Android：Hiddify / v2rayNG

写法要求：
- 不展开过多客户端高级配置
- 只写最短可用路径
- 说明“这条 `vless://` 链接是你后面所有设备共用的核心入口”

### 第五部分：怎么验证自己真的搭成功了

职责：
- 把“验证”单独做成章节，而不是散在文中

至少包含 3 个验证动作：
- mihomo 通过住宅代理访问 `ifconfig.me`
- 外部设备连接 Xray 后普通流量返回 VPS IP
- 访问 Anthropic 域名时，mihomo 日志能看到命中 PROXY

设计要求：
- 每个验证动作都要写“正常结果长什么样”
- 尽量少写“理论上”

### 第六部分：第一次最容易卡住的地方

职责：
- 给普通用户一个最实用的排障入口

优先保留这几类问题：
- 443 没监听
- 7890 没监听
- Google 能开但 Anthropic 不通
- Anthropic 返回 403 / 451
- 客户端导入后连不上

写法要求：
- 问题 -> 最优先检查点 -> 对应命令
- 不堆大量边缘 case

### 第七部分：最后给读者一个维护动作清单

职责：
- 收尾并降低后续维护焦虑

建议保留：
- 启动 / 停止 / 重启命令
- 查看状态命令
- 更换住宅代理后的最短验证动作

结尾要回收一个行动建议：

`如果你只是偶尔用 Claude，这套复杂度未必值得；但如果你已经开始高频依赖它，这篇文章的价值就在于帮你把第一次搭建这件事走顺。`

## 风险与边界

1. 这篇是公众号教程，不是完整运维文档。
   不追求覆盖所有代理商、所有客户端、所有地区差异。

2. 这篇默认保留必要技术细节，但不承诺“零基础零失败”。
   重点是把首次搭建路径讲顺，而不是假装完全无门槛。

3. 这篇不鼓励所有人都去搭。
   文末要提醒：这是用复杂度换稳定性，更适合高频用户。

4. 这篇不直接搬运原稿。
   原始手写教程作为信息源，公众号成稿需要重排结构、重写解释和重做配图。

## 实施备注

真正写稿时，优先参考仓库里 `wechat-openclaw-install/article.md` 的教程节奏：

- 标题先给“新手容易卡的点”
- 开头先给结论和收益
- 正文严格按顺序走
- 每一步都告诉读者“这一步之后你该看到什么”

这篇 Claude 通道文章的差异在于：

- 技术门槛更高，所以要更强调准备项和验证项
- 配图更偏结构图和流程图，而不是操作截图
